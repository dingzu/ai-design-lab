💡核心观点

工程化仍是基础，Token 驱动的设计系统是拥抱 AI 的必要前提。AI 时代下，Token 命名需兼顾「机器可读性」与「人类可理解性」——语义越精确，AI 生成的还原度越高、发散越少。

背景与挑战

  • 一致性规模化需求：多端、多平台、多品牌下，手工维护设计一致性成本急剧上升，需要系统化约束机制
  • 设计与开发协同瓶颈：设计稿到代码的传递链路中存在大量沟通损耗与还原偏差，Token 机制可将设计决策直接以代码形式交付
  • AI 的冲击与赋能：AI 工具爆发既对传统设计生产流程带来冲击，也为自动化生成、智能检索、质量检查提供了新路径；但 AI 能否精准调用设计系统，高度依赖 Token 的语义质量

第一部分 · 工程化基础

  • Design Token（设计令牌）是将原子级设计决策（颜色、字体、间距、圆角等）转化为机器可读、可复用的标准化数据片段，解决「设计与开发不一致」「重复劳动」「系统迭代低效」三大核心问题
  • Token 命名遵循三段式结构 类型_语义_状态，例如 fill_brand_hover 语义为：「填充色」用于「品牌色」的「悬浮」状态

Token 三级层次结构

Lv 1
基础色板 Token（Base Tokens）
完整的基础色板，包含中性色、品牌色、功能色等，是主题的「原始颜料」。如 color-neutral-100 ~ color-neutral-950
Lv 2
语义化主题 Token（Semantic Tokens）
基于 Base Tokens 定义与语义强相关的 token，亮/暗色主题时映射到不同的基础色值，驱动全局主题切换
Lv 3
组件 Token（Component Tokens）
针对基础组件、业务组件基于语义化 token 进一步封装，避免重复定义，提升组件内聚性

Light / Dark 主题 Token 映射示例

  • 语义化 Token 的核心价值在于：切换主题时，只需整体切换变量集，无需逐一修改每个组件，实现一次绑定、全局响应
  • 目前 Figma 尚不支持一键生成并自动绑定 Token，也不内置暗色色值生成功能，需通过手动绑定或借助 AI 工具生成色阶
// 亮色主题 token（映射到浅色基础色板）
lightTokens = {
  // 文本
  'text-primary':  'color-neutral-900',   // 主要文本（深色）
  'text-disabled': 'color-neutral-400',   // 禁用文本（浅灰）
  // 背景
  'bg-page':       'color-neutral-50',    // 页面背景（白色）
  'bg-card':       'color-neutral-100',   // 卡片背景（浅灰）
}

// 暗色主题 token（映射到深色基础色板，逻辑镜像）
darkTokens = {
  'text-primary':  'color-neutral-50',    // 主要文本（白色）
  'text-disabled': 'color-neutral-600',   // 禁用文本（中深灰）
  'bg-page':       'color-neutral-950',   // 页面背景（近黑）
  'bg-card':       'color-neutral-900',   // 卡片背景（深灰）
}

Figma 中的 Token 工作流

四步交付流程

1
创建 Token 基础集
基于自定义基础色阶生成 100–1000 明度色阶，逐一手动绑定 token（Figma 暂不支持一键批量绑定）；暗色色值可借助 AI 工具辅助生成
2
发布至团队组件库
将 Token 集发布至团队，供其他项目引用；在项目中添加该组件库后即可复用全局 token 变量
3
构建组件与 Token 的关联
将组件中的颜色、字体等属性绑定到对应 token 变量；切换变量合集（如切换为 dark 模式）即可全局响应
4
导出 JSON Token 包交付前端
Figma 目前无内嵌导出功能，可通过 Ant Design 主题色编辑器将色值打包为标准 JSON token 文件，发给前端替换即可完成工程交付

第二部分 · AI 时代 Token 命名原则

  • 传统 Token 命名（如 B500、P600、R300）过于抽象，AI 无法准确解析语义，导致代码生成时还原度低、发散率高
  • 好的命名就像餐厅点餐：「我要一份五分熟而非七分熟的牛肉」比「我要牛肉」更精确——Token 的作用正是把设计意图说清楚，减少 AI 的猜测空间

命名对比

❌ 不建议 B500 P600-hover color-2 dis-btn min-padding
✅ AI 时代建议 fill_brand_default fill_primary_hover bg_card_default fill_interactive_disabled spacing_component_minimum

三条命名原则

①
层级分明，三级结构
基础色板 → 语义化 token → 组件 token，层级清晰便于 AI 解析复用层次关系
②
避免歧义，使用机器友好词汇
用 success 而非 good；用 hover 而非 mouse-over；用 disabled 而非 dis；用 minimum 而非 min（AI 可能将 min 混淆为 minute）
③
状态词统一，建立固定词汇库
交互状态固定为 hover / active / disabled；主题变体固定为 light / dark；形成团队内部标准词汇表，AI 可精准识别

场景设想：当设计师输入「为新产品生成一个充满活力的错误状态提示框」，AI 能精准识别出「错误状态」对应 border_error、background_error，「充满活力」映射到更高色彩饱和度——最终输出完全符合设计系统规范的组件，而非随意发挥。

第三部分 · AI 时代设计系统的演进

  • AI 工具的介入并未颠覆设计系统「标准化、可复用、一致性」的核心目标，而是重构了从设计到开发的全链路流程，让工程化落地更高效、更灵活

AI 在设计系统中的三类应用

✦ 智能生成与补全 根据基础 Token 自动生成完整和谐主题；根据设计意图生成组件变体，设计师校准即用
✓ 智能质量保障 AI 代码助手（如 GitHub Copilot）确保开发代码与设计系统一致；AI 设计助手自动检查设计稿是否符合 Token 规范
⌕ 智能检索与推荐 开发者/设计师用自然语言描述需求，AI 从设计系统中推荐最合适的组件和 token

设计系统团队的新职责

  • 从「创建」转向「治理」：重点不再是手动创建每一个组件，而是治理 AI 生成内容的质量，建立准入标准和审核机制
  • 聚焦高层级创新：将基础、重复性工作交给 AI，设计系统团队把精力投入到体验策略、设计语言探索、系统架构决策等更高价值的工作

结语

  • 工程化仍是基础：稳固的、Token 驱动的设计系统是迎接 AI 时代的必要前提，有利于 AI 基于 Design System 生成符合规范的样式
  • 语义设计需面向 AI：Token 和组件的设计需要更具语义性和结构性，以便 AI 理解和精准调用，减少生成时的「幻觉」偏差
  • 设计流程被重塑：AI 将深入设计系统的构建、使用和维护全流程，极大提升设计团队效率——这不是威胁，而是设计师进化的契机

Token 体系越完善、命名越语义化，AI 生成的设计越可控、还原度越高——设计工程化的价值，在 AI 时代非但没有削弱，反而成为了更重要的基础设施。
